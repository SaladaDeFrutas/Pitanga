Processo de manutenção

1) Propósito 

 

Este processo de manutenção possui o intento de controlar evoluções pertinentes ao sistema, promovendo sua melhoria, otimização e o repar de defeitos encontrados. Por meio de determinadas atividades atreladas ao ciclo de vida do software, procura-se formar um processo que garanta alta longevidade.

A organização deverá estar ciente e assegurar conformidade com os seguintes objetivos:

  • Garantir alta manutenibilidade de forma a diminuir os custos com manutenção;
  • Definir a forma como as solicitações de mudança no software devem ocorrer, para manter a qualidade e o controle dos produtos de software da empresa.
  • Assegurar que evoluções (novas versões) do software estão sempre atendendo às necessidades do cliente para assim prolongar a vida útil do software.

 

2) Definições

 

Os seguintes termos e suas definições usadas neste processo são listados abaixo (definições de acordo com a norma ISO/IEC IEE 14764).

        2.01) Manutenção de Melhoria - Uma modificação em um produto de software para satisfazer um novo requisito.

        2.02) Manutenção Corretiva - A modificação reativa (após o relato do problema) em um produto de software realizada após a entrega para corrigir problemas descobertos.

       2.03) Manutenção Preventiva - A modificação em um produto de software, após a entrega, para detectar e corrigir defeitos latentes (que estão na iminência de causar falhas).

        2.04) Manutenibilidade - A capacidade de um produto de software de ser modificado. Modificações incluem correções, melhorias, mudanças no ambiente, mudanças em requisitos e em especificações funcionais (modelos de projeto).

        2.05) Solicitação de Mudança - Documento para as modificações propostas à um produto de software que está sendo mantido.

        2.06) Relatório de Problema - Documento para identificar e descrever problemas detectados em um produto de software que está sendo mantido.

        2.07) Análise de Impacto da Mudança - É a análise realizada para descobrir quais componentes/partes do sistema uma mudança afetará e como essa mudança irá afetar esses componentes.

        2.08) Release - É uma versão aprovada e revisada do sistema que está disponível para uso.

        2.09) Replicação - Consiste na reprodução de um problema, na forma como ele foi detectado. Significa conseguir repetir o problema de forma controlada, identificando a causa desse problema.

        2.10) Transição - É definida como o ato de mudança do software do ambiente de desenvolvimento para o ambiente de manutenção.

 

3) Escopo de Aplicação 

 

        O processo irá agir na área de desenvolvimento de produto de software, de modo a apoiar a sua existência no ambiente do usuário. Isto é, a partir do momento que o desenvolvimento de um produto de software é iniciado, tem-se a ação da manutenção de software, para acompanhar sua evolução.

Após a entrega de uma release, o processo de manutenção irá agir em funcionalidades do software que apresentem problemas para o usuário ou que precisam ser modificadas, após análises prévias destas.

 

4) Políticas

 

 As seguintes políticas serão seguidas pelos envolvidos com o Processo de Manutenção:

  1. Convenção de código fonte. O desenvolvimento de código para implementação de uma modificação, assim como o código anterior à modificação, deve seguir as convenções de codificação da linguagem de programação utilizada (por exemplo, java code conventions), e ser corretamente comentado.
  2. Utilização de padrões de projeto. Modificações em códigos-fonte que foram desenvolvidos e estão organizados em padrões de projeto detalhado devem, se possível, respeitar as estruturas dos padrões para continuar a implementá-los.
  3. Atualização de Documentação. Qualquer documentação pré-existente que for afetada durante a execução do Processo de Manutenção de Software deve ser corretamente atualizada através do repositório utilizado e uma ferramenta de controle de versão(por exemplo, via Git) durante o projeto em aberto. Novos documentos gerados também precisam entrar sobre o controle da gerência de configuração.

 

5) Métricas 

 

Aqui serão definidos as formas para medir o quão bem o processo de manutenção foi executado.

  • O número de erros encontrados com o uso do plugin Checkstyle para o Eclipse;
  • Quantidade de tempo gasto pelos mantenedores durante as etapas de entendimento e análise do produto de software a ser alterado;
  • Quantidade de problemas encontrados derivados de uma determinada mudança;
  • Quantidade de componentes afetados pela mudança identificada em um determinado componente;

7) Definição de Atividades 

 

 

Atividade

Planejar Manutenção

Responsabilidades

Realização: Líder da equipe de manutenção.

Aprovação: Equipe de Manutenção e Equipe de Desenvolvimento.

Colaboração: Membros da equipe de manutenção e Gerente de Projeto.

Informação: Desenvolvedores, equipe de manutenção, gerente de projeto, comitê de gerência de configuração.

Tarefas

1. Definir os responsáveis pela manutenção do produto.

2. Conduzir a análise dos recursos disponíveis para a manutenção.

3. Estimar custos da manutenção e definir cronograma.

4. Verificar a documentação, modelos e código do sistema para  avaliar a manutenibilidade.

5. Determinar os marcos e requisitos de transição.

6. Estabelecer procedimentos para receber, gravar e prover feedback sobre Relatórios de Problemas(RP) e Solicitações de Mudança (SM). O procedimento para esta tarefa está aqui.

Pré-Condições

Projeto Iniciado.

Entradas

-Baselines relevantes para a manutenção;

-Documentação do sistema;

Critérios de Saída

O Plano de Manutenção com os seus procedimentos associados foi feito e aprovado.

Produtos

-Plano de Manutenção;

Infraestrutura

-Uma sala de reunião para o Comitê poder debater sobre as mudanças, com cadeiras e mesa para os membros, caso necessário reuniões podem ser feitas de forma online;

-Computadores necessários para os ambientes utilizados e para os membros do comitê.

Ferramentas

-Um software de controle de versão e de configuração definido, como por exemplo o Git ou o SVN;

-Ferramentas de edição de texto para os vários documentos utilizados e modificados, como Microsoft Word ou LibreOffice Writer;

-Para comunicação, pode ser utilizada uma lista de e-mails ou o ambiente Hangout para reuniões online.

Atividade

Realizar Migração

Responsabilidades

Realização: Líder da equipe de manutenção.

Aprovação: Clientes.

Colaboração: Membros da equipe de manutenção.

Informação: Clientes e usuários do sistema em migração. Usuários do sistema em migração incluem outros sistemas.

Tarefas

1. Identificar todos os produtos de software ou dados que foram adicionados ou modificados dentre um software.

2. Um plano de migração deve ser desenvolvido, documentado e executado, cujo planejamento deve incluir os usuários do sistema.

3. Os usuários do sistema devem ser notificados do planejamento da migração e das atividades geradas pela migração antes dela acontecer.

4. As operações de implementação podem ser feitas em paralelo para garantir uma mudança para o novo sistema mais suave, sendo que o treinamento necessário para os usuários do novo sistema deve ser providenciado como especificado no planejamento.

5. Quando a migração planejada for iniciada, uma notificação deve ser enviada a todos os que serão afetados pela mesma, com os dados do antigo sistema, como documentos, logs e código, devem ser arquivados.

6. Uma revisão de operação deve ser feita para calcular o impacto da mudança de ambiente, sendo que os seus resultados devem ser enviados para as autoridades interessadas para informação, e para nortear as ações feitas pelos mesmos.

7. Os dados associados ao sistema anterior devem ser acessíveis de acordo com os requisitos contratuais, para proteção dos dados e de possível audição dos dados.

Pré-Condições

A Migração foi aprovada e analisada pelos interessados, e os usuários foram avisados da mudança.

Entradas

-O ambiente antigo;

-O novo ambiente;

-A antiga baseline;

-A nova baseline;

Critérios de Saída

A migração foi aceita, planejada e uma notificação da migração foi enviada aos usuários atuais do sistema do que irá acontecer, bem como os dados arquivados do software anterior serão guardados.

Produtos

-Plano de Migração;

-Ferramentas de migração;

-Notificação da Mudança;

-Software Migrado;

-Notificação de migração;

-Dados arquivados.

Infraestrutura

-Uma sala de reunião para o Comitê poder debater sobre as mudanças, com cadeiras e mesa para os membros, caso necessário reuniões podem ser feitas de forma online;

-Computadores necessários para os ambientes utilizados e para os membros do comitê.

Ferramentas

-Um software de controle de versão e de configuração definido, como por exemplo o Git ou o SVN;

-Ferramentas de edição de texto para os vários documentos utilizados e modificados, como Microsoft Word ou LibreOffice Writer;

-Para comunicação, pode ser utilizada uma lista de e-mails ou o ambiente Hangout para reuniões online.
